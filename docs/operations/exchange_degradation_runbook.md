# Runbook: Деградация биржи (Exchange Degradation)

## Обзор

Описывает действия при деградации работы биржи — высокая латентность,
шторм отклонений ордеров, частичная недоступность API.

---

## 1. Симптомы

### Высокая латентность

- Метрика `tb_exchange_latency_ms` > 2000 мс
- Лог: `"Высокая задержка биржи"` (уровень warn)
- Увеличение времени исполнения ордеров

### Reject Storm (шторм отклонений)

- Метрика `tb_order_reject_rate` > 30%
- Лог: `"Массовое отклонение ордеров"` (уровень error)
- Ордера возвращаются с ошибками rate limit или overloaded

### Частичная недоступность

- REST API работает, WebSocket нестабилен (или наоборот)
- Отдельные эндпоинты возвращают 503/429

---

## 2. Автоматическая реакция

### Adversarial Defense

- `ExchangeSlowdown` детектор — снижение уверенности
- `RejectStorm` детектор — блокировка торговли
- При > 5 reject подряд — cooldown 60 секунд

### Risk Guard

- Увеличение минимального интервала между ордерами
- Снижение размера позиции при высокой латентности
- Полная блокировка при критической задержке (> 5000 мс)

### Supervisor

- Переход в деградированный режим при устойчивых проблемах
- Логирование всех инцидентов для post-mortem анализа

---

## 3. Ручные действия

### Диагностика

```bash
# Статус подключения к бирже
curl http://localhost:9090/health/subsystems | jq '.exchange'

# Метрики латентности
curl -s http://localhost:9090/metrics | grep tb_exchange_latency

# Логи ошибок биржи
journalctl -u tomorrow-bot | grep -i "reject\|timeout\|exchange"
```

### Действия оператора

1. Проверить статус биржи: `https://status.bitget.com`
2. При шторме отклонений — активировать safe mode
3. При полной недоступности — kill switch
4. Зафиксировать инцидент для анализа

### Эскалация

- Если проблема > 15 минут — уведомить команду
- Если проблема > 1 час — рассмотреть альтернативные площадки

---

## 4. Восстановление

### Проверка готовности биржи

```bash
# Тестовый запрос к API
curl -s https://api.bitget.com/api/v2/spot/market/tickers | head -c 200
```

### Возобновление торговли

1. Убедиться что латентность нормализовалась (< 500 мс)
2. Проверить что reject rate < 5%
3. Выйти из safe mode
4. Мониторить первые 5 минут активной торговли

### Чеклист после восстановления

- [ ] Латентность < 500 мс стабильно 5 минут
- [ ] Reject rate < 5%
- [ ] Все подсистемы Healthy
- [ ] Стакан синхронизирован
- [ ] Позиции согласованы

---

## 5. Профилактика

- Мониторинг тренда латентности (рост до проблемы)
- Rate limiter на стороне клиента
- Exponential backoff при ошибках
- Регулярная проверка статуса биржи
