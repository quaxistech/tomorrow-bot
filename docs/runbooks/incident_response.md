# Руководство по реагированию на инциденты

## Уровни серьёзности

| Уровень | Описание | Время реакции |
|---------|----------|--------------|
| P0 — Критический | Потеря средств, компрометация безопасности | Немедленно |
| P1 — Высокий | Система не торгует, большие убытки | 5 минут |
| P2 — Средний | Деградация производительности | 15 минут |
| P3 — Низкий | Ошибки без влияния на торговлю | 1 час |

## P0: Критический инцидент — потеря средств

### Немедленные действия (первые 2 минуты)
1. **ОСТАНОВИТЬ СИСТЕМУ**: `sudo systemctl stop tomorrow-bot`
2. Зайти в интерфейс биржи, отменить все открытые ордера вручную
3. Задокументировать время и обстоятельства
4. Уведомить команду

### Расследование
1. Собрать логи: `journalctl -u tomorrow-bot -n 1000 > incident.log`
2. Проверить метрики за последний час
3. Найти первую аномалию в логах
4. Определить root cause

## P1: Система не торгует

### Диагностика
```bash
# Проверить статус процесса
systemctl status tomorrow-bot

# Проверить health
curl -f http://localhost:8080/health || echo "Health endpoint недоступен"

# Проверить логи
journalctl -u tomorrow-bot --since "10 minutes ago" -p err
```

### Распространённые причины

| Симптом | Вероятная причина | Действие |
|---------|------------------|----------|
| `ExchangeConnectionFailed` | Сеть/API биржи | Проверить connectivity |
| `ExchangeAuthFailed` | Ротация API ключей | Обновить ключи |
| `MarketDataStale` | Разрыв WebSocket | Перезапустить |
| `ConfigValidationFailed` | Неверная конфигурация | Проверить configs/ |
