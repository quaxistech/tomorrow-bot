# Модуль app

## Что это за блок

`src/app` — это точка входа и orchestration-слой всего приложения. Этот блок сам почти не содержит торговой логики: он не принимает торговые решения, не рассчитывает сигналы и не исполняет ордера напрямую. Его задача другая: собрать инфраструктуру, связать между собой крупные подсистемы, запустить процесс и корректно его остановить.

Если смотреть на проект сверху, то `app` играет роль "composition root". Именно здесь приложение:

- читает аргументы командной строки;
- загружает конфигурацию;
- получает секреты и проверяет, можно ли вообще запускать бота;
- поднимает логирование, метрики и системные часы;
- запускает HTTP endpoint для Prometheus-метрик;
- создаёт сканер рынка;
- выбирает набор торговых символов;
- создаёт по одному `TradingPipeline` на каждый активный символ;
- регистрирует эти pipeline в `Supervisor`;
- управляет ротацией символов и shutdown-сценарием.

Иначе говоря, `app` отвечает не за "как торговать", а за "как собрать и держать работающей всю торговую систему".

## Какие файлы входят в модуль

### `main.cpp`

Это реальная точка входа процесса. Здесь находится `main()`, и именно отсюда начинается запуск всего бота.

Основные обязанности файла:

- разобрать CLI-аргументы (`--config`, `--help`, `--version`);
- вызвать bootstrap приложения;
- вывести стартовый баннер в лог;
- при необходимости поднять HTTP-сервер метрик;
- создать и запустить `ScannerEngine`;
- определить итоговый список символов для торговли;
- проверить уже открытые позиции на бирже и принудительно добавить их символы в список отслеживаемых;
- создать `Supervisor`;
- собрать `TradingPipeline` для каждого символа;
- включить периодическую ротацию символов и отдельный idle-monitor для рескана рынка;
- дождаться сигнала завершения;
- остановить фоновые сервисы и pipeline.

Это самый важный файл в блоке, потому что он задаёт полный жизненный цикл процесса от старта до остановки.

### `app_bootstrap.hpp` / `app_bootstrap.cpp`

Это слой начальной инициализации инфраструктуры.

`AppBootstrap` собирает объект `AppComponents`, в котором находятся базовые зависимости верхнего уровня:

- `config_loader`;
- `secret_provider`;
- `logger`;
- `metrics`;
- `clock`;
- итоговый `config`.

Bootstrap выполняет запуск в правильной последовательности:

1. Загружает конфигурацию через модуль `config`.
2. Ищет `.env` и выбирает источник секретов через модуль `security`.
3. Настраивает логгер через модуль `logging`.
4. Создаёт реестр метрик через модуль `metrics`.
5. Создаёт системные часы через модуль `clock`.
6. Запускает `ProductionGuard`, который проверяет, допустим ли старт процесса в текущем торговом режиме.

Смысл этого слоя в том, чтобы `main.cpp` не занимался низкоуровневой инициализацией каждого сервиса вручную. Вместо этого `main.cpp` получает уже собранный комплект системных зависимостей.

### `http_server.hpp` / `http_server.cpp`

Это маленький встроенный HTTP-сервер на Boost.Asio/Boost.Beast.

Он нужен не для пользовательского веб-интерфейса и не для API управления ботом. Его задача значительно уже: отдать диагностические endpoint'ы, прежде всего Prometheus-метрики.

Особенности сервера:

- сервер запускается в отдельном потоке;
- принимает HTTP-запросы синхронно внутри собственного `io_context`;
- обрабатывает каждый запрос через callback `HttpRequestHandler`;
- по текущей конфигурации слушает loopback-адрес `127.0.0.1`, то есть ориентирован на локальный мониторинг, а не на внешний публичный доступ;
- в `main.cpp` используется для выдачи `metrics->export_prometheus()`.

Это инфраструктурный адаптер между внутренним реестром метрик и внешним monitoring stack.

### `CMakeLists.txt`

Файл сборки модуля `app`.

Он собирает исполняемый файл `tomorrow-bot` из:

- `main.cpp`;
- `app_bootstrap.cpp`;
- `http_server.cpp`.

Также он связывает исполняемый файл с крупными библиотеками проекта:

- `tb_pipeline`;
- `tb_scanner`;
- `tb_exchange_bitget`;
- `tb_config`;
- `tb_security`;
- `tb_logging`;
- `tb_metrics`;
- `tb_clock`;
- `tb_supervisor`;
- Boost и Threads.

То есть по CMake хорошо видно, что `app` — это именно верхний слой композиции, который зависит почти от всех ключевых модулей, но сам остаётся сравнительно тонким.

## Что делает модуль app по шагам

Ниже логика работы блока в реальном порядке исполнения.

### 1. Принимает параметры запуска

Сначала `main.cpp` читает аргументы командной строки.

Поддерживаются три основных сценария:

- показать помощь;
- показать версию;
- запустить процесс с указанным конфигом.

Если путь до конфига не передан, используется значение по умолчанию: `configs/production.yaml`.

### 2. Инициализирует базовую инфраструктуру

Дальше управление передаётся в `AppBootstrap`.

В этот момент создаётся основа всей системы:

- конфигурация проекта;
- доступ к секретам API;
- логирование;
- метрики;
- часы;
- pre-flight проверка режима запуска.

Здесь важно понимать: если bootstrap не прошёл, приложение вообще не стартует дальше. Значит `app` — это первый защитный рубеж перед тем, как бот начнёт говорить с биржей или строить pipeline.

### 3. Настраивает мониторинг

Если метрики включены в конфиге, `main.cpp` поднимает `HttpEndpointServer`.

Handler этого сервера очень простой:

- принимает только `GET`;
- проверяет, что запрос пришёл по ожидаемому пути (`config.metrics.path`);
- возвращает 404 на другие пути;
- возвращает 405 на неподдерживаемые методы;
- на корректный запрос отдаёт текст метрик в формате Prometheus.

То есть `app` интегрирует внутреннюю систему наблюдаемости с внешним мониторингом.

### 4. Выбирает рынки для торговли

После этого `main.cpp` создаёт публичный `BitgetRestClient` и через него поднимает `ScannerEngine`.

`app` сам не вычисляет скоринг активов. Это делает модуль `scanner`. Но именно `app`:

- собирает `ScannerConfig` из общей конфигурации приложения;
- ослабляет некоторые фильтры для микро-аккаунтов;
- запускает первичное сканирование;
- получает итоговый `ScannerResult`;
- извлекает оттуда список символов для торговли.

Если сканер ничего не нашёл, `app` включает fallback-логику и выбирает символ из `pair_selection.manual_symbols`, а если список пуст — использует запасной символ.

Это важный момент: модуль `app` принимает организационное решение, на каких именно инструментах запускать торговые pipeline в текущем процессе.

### 5. Проверяет уже открытые позиции на бирже

Перед созданием pipeline `main.cpp` делает ещё один важный шаг: запрашивает у Bitget список уже открытых фьючерсных позиций.

Зачем это нужно:

- чтобы после рестарта процесса не потерять символ, по которому уже есть живая позиция;
- чтобы не запустить систему только на "новых" сканером выбранных парах и забыть про риск на старых;
- чтобы логировать текущее состояние биржи при старте.

Если у биржи есть открытая позиция по символу, которого нет в списке `active_symbols`, `app` принудительно добавляет этот символ в список отслеживаемых.

То есть `app` обеспечивает связность между состоянием процесса и реальным состоянием аккаунта на бирже.

### 6. Собирает правила инструментов

По результатам сканирования `app` вытягивает precision и минимальные торговые ограничения по инструментам и складывает их в `symbol_rules`.

Эти данные потом передаются в каждый `TradingPipeline`, чтобы pipeline умел:

- округлять количество и цену;
- учитывать минимальный notional;
- корректно взаимодействовать с биржевыми ограничениями.

Это уже не просто выбор символов, а ещё и передача pipeline технической спецификации по каждому инструменту.

### 7. Создаёт Supervisor

Дальше `main.cpp` создаёт объект `tb::supervisor::Supervisor`.

Это центральный lifecycle-координатор системы. `app` использует его как диспетчер запуска и остановки подсистем.

На этом этапе `app`:

- передаёт `Supervisor` логгер, метрики и часы;
- устанавливает signal handlers;
- позже регистрирует в нём каждый pipeline как отдельную управляемую подсистему.

Здесь видно архитектурное разделение ролей:

- `app` решает, какие подсистемы должны существовать;
- `Supervisor` отвечает за то, как ими управлять в runtime.

### 8. Создаёт TradingPipeline для каждого символа

Это основная точка передачи управления из orchestration-слоя в торговую логику.

Для каждого символа `app` создаёт `tb::pipeline::TradingPipeline`, передавая туда:

- общий `config`;
- `secret_provider`;
- `logger`;
- `clock`;
- `metrics`;
- сам символ.

После этого `app` ещё дополнительно конфигурирует pipeline:

- устанавливает exchange rules, если они были получены из сканера;
- сообщает количество параллельных pipeline, чтобы капитал можно было корректно делить между символами;
- регистрирует пары функций start/stop в `Supervisor`.

Именно в этот момент `app` превращает абстрактную конфигурацию и список символов в реальные рабочие торговые контуры.

### 9. Запускает периодическую ротацию символов

У `ScannerEngine` включается фоновая ротация: через заданный интервал сканер снова оценивает рынок и логирует лучшие пары.

Важно: текущая callback-логика в `main.cpp` пока только логирует результат плановой ротации. Саму горячую замену pipeline выполняет не эта часть, а отдельный idle-monitor.

То есть `app` использует два разных механизма:

- плановая ротация как информационный/наблюдаемый процесс;
- фактическая пересборка pipeline при долгом простое.

### 10. Запускает idle-monitor и горячую замену pipeline

После старта `Supervisor` модуль `app` создаёт отдельный поток, который периодически проверяет состояние всех pipeline.

Логика этого потока такая:

- ждать заданный интервал;
- не делать рескан слишком часто;
- проверить, что все pipeline простаивают дольше порога;
- убедиться, что ни у одного pipeline нет открытой позиции;
- если рынок текущими парами не даёт активности, выполнить повторное сканирование;
- если новые символы отличаются от старых, остановить старые pipeline и собрать новые.

Это одна из самых важных функций блока `app`: он не только один раз запускает систему, но и умеет на уровне orchestration динамически заменить активные торговые контуры без перезапуска всего процесса.

### 11. Держит процесс живым до сигнала завершения

После запуска всех подсистем `app` переходит в режим ожидания через `supervisor.wait_for_shutdown()`.

С этого момента основная работа происходит уже внутри pipeline, market data, strategy, risk и execution модулей. Но `app` остаётся управляющей оболочкой процесса, которая ждёт операционного сигнала на завершение.

### 12. Корректно завершает работу

Когда приходит сигнал остановки, `app` делает coordinated shutdown:

- останавливает поток idle-monitor;
- останавливает rotation у сканера;
- останавливает HTTP-сервер метрик;
- вручную вызывает `stop()` у каждого pipeline;
- вызывает `supervisor.stop()`;
- пишет финальное сообщение в лог.

Это означает, что `app` отвечает ещё и за безопасное завершение процесса, а не только за старт.

## С чем взаимодействует модуль app

Ниже перечислены его ключевые зависимости и характер взаимодействия.

### `config`

Используется для загрузки и хранения полной конфигурации приложения.

`app` зависит от `config` практически на каждом шаге:

- режим торговли;
- логирование;
- endpoint метрик;
- настройки сканера;
- параметры выбора торговых пар;
- REST endpoint биржи;
- futures-конфигурация.

Без `config` модуль `app` не может ни собрать инфраструктуру, ни выбрать режим работы.

### `security`

Используется в двух ролях:

- как поставщик секретов (`ISecretProvider`);
- как pre-flight guard (`ProductionGuard`).

Через эту зависимость `app` получает API-ключи, секрет и passphrase, а также проверяет, разрешён ли запуск процесса в production/paper-режиме.

То есть `app` взаимодействует с `security` не только ради доступа к секретам, но и как с механизмом контроля запуска.

### `logging`

Это основной канал наблюдаемости за жизненным циклом процесса.

`app` активно логирует:

- старт бота;
- параметры запуска;
- запуск endpoint метрик;
- результаты сканирования;
- список активных символов;
- создание pipeline;
- события ротации;
- проблемы доступа к бирже;
- процедуры shutdown.

По сути, весь процесс orchestration прозрачен именно благодаря модулю логирования.

### `metrics`

`app` не считает торговые метрики сам, но он:

- создаёт общий реестр метрик;
- передаёт его вниз в другие подсистемы;
- поднимает HTTP endpoint для экспорта этих метрик наружу.

Значит именно `app` замыкает внутренний сбор метрик на внешний monitoring-интерфейс.

### `clock`

Используется как единый источник времени для:

- startup-компонентов;
- supervisor;
- pipeline;
- таймингов idle-monitor и throttling логики.

`app` не реализует временную бизнес-логику, но координирует процессы, для которых время критично.

### `scanner`

Сканер рынка является для `app` сервисом выбора инструментов.

`app`:

- настраивает `ScannerEngine`;
- запускает scan;
- получает кандидатов;
- извлекает метаданные по символам;
- инициирует rotation;
- использует повторное сканирование как триггер для горячей замены pipeline.

То есть `scanner` для `app` — это механизм выбора предмета торговли.

### `exchange::bitget`

`app` напрямую использует Bitget-модуль в двух местах:

- создаёт публичный REST-клиент для сканера;
- создаёт авторизованный REST-клиент и `BitgetFuturesQueryAdapter` для запроса открытых позиций на старте.

Иначе говоря, `app` сам не торгует на бирже, но он синхронизирует стартовую картину мира приложения с биржевой реальностью.

### `pipeline`

Это главная рабочая зависимость модуля `app`.

`TradingPipeline` — место, где уже происходит реальная торговая обработка: market data, features, world model, regime, strategy, risk, execution, reconciliation и так далее.

Роль `app` относительно `pipeline`:

- создать pipeline;
- передать ему общие зависимости;
- сообщить символ;
- передать биржевые правила;
- зарегистрировать lifecycle hooks;
- остановить и пересоздать pipeline при ротации.

Поэтому `app` можно описать как слой, который управляет множеством торговых pipeline, но не вмешивается глубоко в их внутренние вычисления.

### `supervisor`

`Supervisor` даёт `app` стандартный механизм управления подсистемами.

Через него `app`:

- регистрирует pipeline как подсистемы;
- запускает их в общем жизненном цикле;
- ждёт сигнал завершения;
- централизованно инициирует shutdown.

Это делает `app` не просто procedural main-файлом, а точкой построения управляемой системы с lifecycle semantics.

## Чего модуль app не делает

Для понимания архитектуры важно явно сказать, что этот блок НЕ делает.

`app` не:

- рассчитывает индикаторы;
- не строит order book-модели;
- не выбирает направление сделки;
- не принимает risk-решение;
- не вычисляет плечо;
- не исполняет ордер;
- не ведёт reconciliation ордеров и позиций.

Все эти задачи делегируются в более глубокие модули, в первую очередь в `TradingPipeline` и его зависимости.

То есть `app` — это не торговый движок, а модуль запуска, сборки и операционного управления торговым движком.

## Архитектурная роль блока

С архитектурной точки зрения `src/app` выполняет сразу несколько ролей.

### 1. Точка входа процесса

Без этого блока бинарник просто не существовал бы как исполняемое приложение.

### 2. Composition Root

Именно здесь конкретные реализации сервисов соединяются друг с другом. До этого уровня проект состоит из библиотечных модулей; здесь они превращаются в работающее приложение.

### 3. Lifecycle Orchestrator

`app` управляет полным жизненным циклом процесса: startup, runtime orchestration, rotation, shutdown.

### 4. Boundary Layer

Через этот слой приложение взаимодействует с внешним окружением:

- CLI;
- конфиги;
- `.env`/переменные окружения;
- мониторинг;
- сигналы ОС;
- биржа на старте;
- операционные сценарии рестарта и остановки.

## Практический вывод

Если формулировать совсем коротко, модуль `app` делает следующее:

`app` собирает из набора библиотечных модулей единый работающий торговый процесс, выбирает для него символы, создаёт торговые pipeline, следит за их жизненным циклом, поддерживает мониторинг и обеспечивает безопасный старт и остановку системы.

Если понадобится менять:

- способ запуска процесса;
- bootstrap-зависимости;
- HTTP endpoint метрик;
- логику выбора начальных символов;
- orchestration нескольких pipeline;
- логику ротации пар;
- shutdown-последовательность,

то основная точка изменений находится именно в `src/app`.
