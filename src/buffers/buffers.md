# Модуль buffers — подробный разбор текущей реализации

Временный рабочий документ.

Дата: 2026-04-09

## 1. Назначение модуля

`src/buffers` — это небольшой, но важный инфраструктурный модуль хранения короткой рыночной истории в памяти.

Он не занимается ни сетевым вводом, ни торговыми решениями, ни вычислением индикаторов сам по себе. Его задача уже и конкретнее:

1. принимать поток последних свечей и сделок;
2. держать только ограниченное окно истории фиксированного размера;
3. позволять быстро извлекать эти данные в виде, удобном для `FeatureEngine`;
4. не разрастаться бесконечно по памяти.

Архитектурно это **не отдельный runtime-сервис**, а header-only библиотека контейнеров и адаптеров поверх одного базового примитива `RingBuffer`.

В production-path модуль buffers фактически обслуживает `FeatureEngine`:

- `CandleBuffer` хранит историю OHLCV для технических признаков;
- `TradeBuffer` хранит последние сделки для микроcтруктурных признаков;
- `RingBuffer` даёт обоим фиксированную ёмкость и overwrite-семантику.

## 2. Состав модуля

В каталоге `src/buffers` лежат:

| Файл | Назначение |
|---|---|
| `ring_buffer.hpp` | базовый шаблонный кольцевой буфер фиксированного размера |
| `candle_buffer.hpp` | специализированный буфер свечей OHLCV |
| `trade_buffer.hpp` | специализированный буфер сделок и простых trade-flow метрик |
| `CMakeLists.txt` | объявление header-only библиотеки `tb_buffers` |

Сборка оформлена как:

- `add_library(tb_buffers INTERFACE)`;
- include path: `${CMAKE_SOURCE_DIR}/src`;
- зависимости: `tb_common`, `tb_normalizer`.

Это означает, что у модуля нет собственных `.cpp`-единиц трансляции и нет отдельного runtime-state вне объектов, создаваемых пользователем.

## 3. Реальная роль buffers в проекте

На текущем состоянии кода `buffers` используется не «по всему проекту», а в очень конкретной цепочке:

1. `MarketDataGateway` и normalizer производят `NormalizedCandle`, `NormalizedTrade`, `NormalizedTicker`;
2. `FeatureEngine::on_candle()` и `FeatureEngine::on_trade()` записывают события в буферы;
3. `FeatureEngine::compute_snapshot()` извлекает из буферов массивы цен/объёмов и считает признаки;
4. эти признаки попадают в pipeline, стратегии, risk и execution.

Иными словами, buffers — это **локальная in-memory history layer для feature computation**.

Важно: по текущему коду у модуля нет прямых пользователей кроме `FeatureEngine`.

## 4. Главный примитив: `RingBuffer<T, Capacity>`

### 4.1. Базовая модель данных

`RingBuffer` хранит:

- `std::array<T, Capacity> data_`;
- `head_` — индекс следующей записи;
- `count_` — текущее число валидных элементов.

Модель работы классическая для overwrite ring buffer:

- пока буфер не заполнен, новые элементы просто дописываются;
- после заполнения новые элементы начинают перезаписывать самые старые.

Это делает поведение детерминированным по памяти:

- размер всегда ограничен `Capacity`;
- история автоматически «скользит» вперёд;
- старые данные не надо чистить вручную.

### 4.2. Как работает `push()`

При `push(T value)` происходит:

1. запись элемента в `data_[head_]`;
2. `head_ = (head_ + 1) % Capacity`;
3. если `count_ < Capacity`, счётчик увеличивается.

Семантически это означает:

- `head_` всегда указывает **не на последний элемент, а на следующую позицию записи**;
- когда буфер полон, позиция `head_` логически соответствует самому старому элементу, который будет перезаписан следующим.

### 4.3. Как определяется порядок элементов

Ключевая функция — `start_index()`:

- если буфер ещё не полный, начало истории считается нулевым индексом;
- если буфер полный, началом считается `head_`, потому что именно там лежит самый старый живой элемент.

Дальше `operator[]` интерпретирует индекс так:

- `0` — самый старый элемент в окне;
- `size() - 1` — самый новый.

Это очень важный контракт для всего модуля:

- `CandleBuffer` и `TradeBuffer` везде опираются именно на порядок «от старых к новым»;
- `FeatureEngine` передаёт индикаторам массивы именно в этой хронологии.

### 4.4. `front()` и `back()`

`front()` и `back()` дают доступ к:

- самому старому элементу;
- самому новому элементу.

На пустом буфере они бросают `std::out_of_range`.

Это более строгий контракт, чем у `operator[]`.

### 4.5. `operator[]` и важная особенность контракта

У `operator[]` нет bounds-check.

То есть класс предполагает, что вызывающий код сам гарантирует:

- `idx < size()`.

Внутренний код модуля buffers это условие соблюдает, но как общий контейнер `RingBuffer` не защищает пользователя от ошибки диапазона.

Следствие:

- это быстрый low-level контейнер;
- он рассчитан на дисциплинированный use-site, а не на defensive API.

### 4.6. Дополнительные методы

Есть несколько полезных служебных методов:

- `size()`;
- `empty()`;
- `full()`;
- `capacity()`;
- `clear()`;
- `to_vector()`;
- `extract_field()`.

Из них в текущем production-path реально используются:

- `push()`;
- `size()`;
- `empty()`;
- `back()`;
- `extract_field()`.

А методы:

- `full()`;
- `capacity()`;
- `clear()`;
- `to_vector()`

сейчас лежат как общая полезная API-поверхность, но по текущему коду прямо не задействованы внешними модулями.

### 4.7. Асимптотика и память

По стоимости операций:

- `push()` — $O(1)$;
- `operator[]` — $O(1)$;
- `front()` / `back()` — $O(1)$;
- `extract_field()` — $O(n)$;
- `to_vector()` — $O(n)$.

После создания самого объекта `RingBuffer` новых аллокаций внутри него не происходит, потому что данные лежат в `std::array` фиксированного размера.

Но это не означает, что весь feature-path без аллокаций:

- при извлечении в `std::vector` аллокации уже происходят в вызывающем слое.

Это важное инженерное различие:

- **сам контейнер zero-allocation after init**;
- **потребляющий слой может быть allocation-heavy**.

### 4.8. Неявные ограничения шаблона

У реализации есть несколько неявных требований к типу `T`:

1. `T` должен быть корректно default-initializable, потому что `std::array<T, Capacity> data_{}` создаётся сразу;
2. `T` должен нормально переживать move/copy assignment в `push()`;
3. желательно, чтобы это был дешёвый по копированию тип.

В текущем модуле это условие выполняется:

- `OhlcvCandle` и `TradeRecord` — простые компактные структуры.

## 5. `CandleBuffer`

### 5.1. Что хранит `CandleBuffer`

`CandleBuffer` — это thin wrapper над `RingBuffer<OhlcvCandle, Capacity>`.

Структура `OhlcvCandle` содержит:

- `open_time`;
- `open`;
- `high`;
- `low`;
- `close`;
- `volume`;
- `is_closed`.

То есть это именно внутренний формат для теханализа, а не универсальная биржевая свеча.

### 5.2. Два пути записи в буфер

`CandleBuffer` умеет принимать:

1. уже готовый `OhlcvCandle`;
2. `normalizer::NormalizedCandle`.

Во втором случае происходит простой mapping:

- `exchange_ts -> open_time`;
- `open/high/low/close/volume` распаковываются из typed wrappers;
- `is_closed` переносится как есть.

Сам буфер не делает ни валидацию свечи, ни нормализацию времени, ни проверку таймфрейма.

### 5.3. Семантика `update_last()`

Это один из самых важных методов модуля.

Поведение:

- если буфер пустой **или последняя свеча уже закрыта** — `push()` (новый период);
- иначе обновляет поля последней live-свечи:
  - `high` → `std::max(old_high, new_high)` (defensive);
  - `low` → `std::min(old_low, new_low)` (defensive);
  - `close`, `volume`, `is_closed` — прямое присвоение.

При этом **не меняются**:

- `open_time`;
- `open`.

Важные решённые проблемы:

1. **Переход между периодами**: проверка `is_closed` последней свечи предотвращает коррупцию закрытой свечи данными нового периода.
2. **Defensive high/low**: `std::max`/`std::min` гарантируют монотонность high и low даже при out-of-order или partial upstream-обновлениях.

### 5.4. Как `push()` обрабатывает закрытие live-свечи

`push(NormalizedCandle)` решает проблему дублирования свечей:

- если приходит `is_closed=true` и последняя свеча в буфере не закрыта — **финализирует её на месте** (заменяет live-версию);
- иначе — добавляет новую запись.

Без этой логики каждое закрытие создавало бы дубль: сталая live-копия + финальная. Это корруптировало бы историю для даунстрим-индикаторов.

### 5.5. Извлечение OHLCV-массивов

`CandleBuffer` умеет возвращать:

- `close_prices()`;
- `high_prices()`;
- `low_prices()`;
- `open_prices()`;
- `volumes()`.

Все методы:

- извлекают последние `n` элементов;
- сохраняют порядок от старых к новым;
- возвращают `std::vector<double>`.

Это именно тот формат, которого ожидает `IndicatorEngine`.

### 5.6. `has_enough()`

`has_enough(required)` — маленький, но практический helper:

- он не знает ничего про индикаторы;
- просто проверяет `buffer_.size() >= required`.

В реальном коде он используется в `FeatureEngine::is_ready()` и в `compute_snapshot()` для gating по длине истории.

## 6. `TradeBuffer`

### 6.1. Что хранит `TradeBuffer`

`TradeBuffer` — это wrapper над `RingBuffer<TradeRecord, Capacity>`.

`TradeRecord` содержит:

- `ts`;
- `price`;
- `size`;
- `side` (`Buy` / `Sell`);
- `is_aggressive`.

То есть буфер хранит уже нормализованные принты сделок в форме, удобной для микроcтруктурной аналитики.

### 6.2. Путь записи

Как и в `CandleBuffer`, есть два варианта записи:

1. `push(TradeRecord)`;
2. `push(NormalizedTrade)`.

Mapping здесь прямой:

- `exchange_ts -> ts`;
- `price.get() -> price`;
- `size.get() -> size`;
- `side` переносится как есть;
- `is_aggressive` переносится как есть.

`TradeBuffer` не агрегирует сделки и не пытается строить bars — он работает на уровне отдельных trade prints.

### 6.3. `vwap(n)`

Метод считает VWAP по последним `n` записям:

$$
VWAP = \frac{\sum price_i \cdot size_i}{\sum size_i}
$$

Поведение на краях:

- если окно пустое, возвращает `0.0`;
- если суммарный объём нулевой, тоже возвращает `0.0`.

Это безопасное поведение для downstream-кода, но важно понимать, что `0.0` здесь означает не «рыночная цена ноль», а «недостаточно содержательного объёма».

### 6.4. `buy_sell_ratio(n)`

Метод считает отношение:

$$
buy\_sell\_ratio = \min\left(\frac{buy\_volume}{sell\_volume}, 100\right)
$$

По edge-case поведению:

- если окно пустое, возвращается `1.0` как нейтральное значение;
- если `sell_vol > 0`, возвращается `min(buy/sell, 100.0)`;
- если `sell_vol == 0` и `buy_vol > 0`, возвращается `100.0` как cap.

Cap = 100.0 предотвращает blow-up в downstream ML-пайплайне и стратегиях. 100:1 уже представляет экстремальный дисбаланс, при этом сигнал не теряется.

### 6.5. `aggressive_flow(n)`

Метод считает долю **агрессивного объёма** (volume-weighted):

$$
aggressive\_flow = \frac{\sum_{aggressive} size_i}{\sum_{all} size_i}
$$

Научное обоснование: Kyle (1985) показывает, что price impact пропорционален размеру сделки, а не количеству сделок. Chordia, Roll & Subrahmanyam (2002) подтверждают, что volume-weighted order flow более предиктивен для краткосрочного price impact.

Edge-case: если объёма нет (пустое окно или нулевой total_volume), возвращается `0.5` — нейтральное значение.

## 7. Как buffers встроен в `FeatureEngine`

## 7.1. Где реально живут буферы

В `FeatureEngine` объявлены:

- `std::unordered_map<std::string, CandleBuffer<500>> candle_buffers_`;
- `std::unordered_map<std::string, TradeBuffer<1000>> trade_buffers_`;
- `std::unordered_map<std::string, NormalizedTicker> last_tickers_`.

То есть буферы сегментируются так:

- свечи — по ключу `symbol:interval`;
- сделки — по ключу `symbol`.

Это важное архитектурное решение:

- `CandleBuffer` сам по себе не знает про таймфрейм;
- сегментация таймфреймов обеспечивается уровнем `FeatureEngine`, а не самим буфером.

### 7.2. `on_candle()`

`FeatureEngine::on_candle()` делает:

1. берёт mutex;
2. строит ключ `symbol:interval`;
3. получает соответствующий `CandleBuffer`;
4. если `candle.is_closed == true`, вызывает `push()`;
5. иначе вызывает `update_last()`.

Отсюда следует ключевая runtime-семантика:

- буфер свечей хранит не только закрытую историю;
- последняя запись может быть live-updating незакрытой свечой.

### 7.3. `on_trade()`

`FeatureEngine::on_trade()`:

1. берёт mutex;
2. берёт ключ `symbol`;
3. пушит нормализованную сделку в `TradeBuffer`.

Здесь модель проще:

- нет mutable last element;
- каждая сделка просто добавляется в скользящее окно.

### 7.4. `is_ready()`

Готовность `FeatureEngine` фактически завязана только на `CandleBuffer` основного интервала.

Требуемое число свечей считается как:

$$
max(sma, ema\_slow, macd\_slow, bb, atr, adx) + 1
$$

На текущих дефолтах это:

- `max(20, 50, 26, 20, 14, 14) + 1 = 51` свеча.

То есть именно `CandleBuffer` определяет, готов ли движок к расчёту технических признаков.

### 7.5. `compute_technical()`

При построении technical features происходит:

1. берётся `CandleBuffer` нужного `symbol:interval`;
2. из него извлекаются векторы:
   - `close`;
   - `high`;
   - `low`;
   - `volume`;
3. эти векторы передаются в `IndicatorEngine`.

Следовательно, CandleBuffer — это не просто storage, а **фактический источник всех базовых time-series для теханализа**.

Через него проходят:

- SMA/EMA;
- RSI;
- MACD;
- Bollinger Bands;
- ATR;
- ADX;
- OBV;
- volatility;
- momentum.

### 7.6. `compute_microstructure()`

Для микроструктуры `TradeBuffer` используется как источник трёх признаков:

- `buy_sell_ratio`;
- `aggressive_flow`;
- `trade_vwap`.

Размер окна задаётся `FeatureEngine::Config::trade_flow_window`.

Иными словами, `TradeBuffer` — это встроенный rolling trade-flow calculator, а не просто сырое хранилище.

## 8. Модель конкурентности

Сами `RingBuffer`, `CandleBuffer` и `TradeBuffer` **не потокобезопасны**.

Они не содержат:

- mutex;
- atomic;
- lock-free примитивов.

Потокобезопасность обеспечивается не внутри них, а на уровне `FeatureEngine`, который сериализует доступ через один `mutable std::mutex mutex_`.

Практическое следствие:

- буферы задуманы как внутренние структуры под внешней блокировкой;
- использовать их concurrently напрямую без синхронизации нельзя.

## 9. Производительность и аллокации

## 9.1. Что здесь сделано хорошо

С точки зрения low-level design модуль очень лёгкий:

- фиксированные capacities;
- простые POD-подобные записи;
- `RingBuffer` не растёт динамически;
- `push()` предсказуем и дешёв.

Для скальперского USDT-M бота это правильное направление: история ограничена, latency и memory footprint контролируемы.

### 9.2. Где всё же происходят аллокации

Несмотря на zero-allocation контейнер, downstream path не полностью zero-allocation.

Причина:

- `CandleBuffer::close_prices()/high_prices()/...` создают новые `std::vector<double>`;
- `FeatureEngine::compute_technical()` строит такие векторы на каждом snapshot.

Итого:

- хранение истории дешёвое;
- извлечение истории для индикаторов уже выделяет память и копирует данные.

Это не делает модуль плохим, но важно честно понимать ограничение:

- **buffers cheap to maintain**;
- **snapshot extraction not zero-copy**.

### 9.3. Работа под mutex

`FeatureEngine::compute_snapshot()` держит общий mutex, пока:

1. проверяет readiness;
2. копирует данные из буферов в векторы;
3. вычисляет признаки.

Это означает, что буферы сами быстрые, но критическая секция выше них может быть длиннее, чем кажется по интерфейсу buffers.

## 10. Что реально используется, а что пока просто есть в API

### 10.1. Реально используется сейчас

- `RingBuffer::push()`;
- `RingBuffer::size()`;
- `RingBuffer::empty()`;
- `RingBuffer::back()`;
- `RingBuffer::extract_field()`;
- `CandleBuffer::push()`;
- `CandleBuffer::update_last()`;
- `CandleBuffer::close_prices()/high_prices()/low_prices()/volumes()`;
- `CandleBuffer::has_enough()`;
- `TradeBuffer::push()`;
- `TradeBuffer::vwap()`;
- `TradeBuffer::buy_sell_ratio()`;
- `TradeBuffer::aggressive_flow()`.

### 10.2. Есть в общей библиотеке, но сейчас не видны в production-path

- `RingBuffer::full()`;
- `RingBuffer::capacity()`;
- `RingBuffer::clear()`;
- `RingBuffer::to_vector()`;
- `RingBuffer::front()`.

Это не ошибка. Скорее это признак того, что `RingBuffer` спроектирован как общий reusable primitive с немного более широкой API, чем нужно текущему `FeatureEngine`.

## 11. Важные инженерные наблюдения

### 11.1. Это не «модуль с логикой», а модуль с контрактами

Внешне buffers выглядит примитивно, но его ценность не в сложности алгоритмов, а в корректности контрактов:

- порядок элементов;
- overwrite-семантика;
- поведение на пустом окне;
- смысл нейтральных default values;
- предположение о полноте входящих candle updates.

Если эти контракты нарушить, downstream-аналитика будет неверной даже при формально корректном коде.

### 11.2. `CandleBuffer` intentionally live-like, а не strictly-close-only

Так как `update_last()` меняет последнюю свечу до её закрытия, технические признаки могут считаться по незакрытому бару.

Это даёт:

- более реактивное live-поведение;
- более ранний сигнал для скальпинга.

Но это также означает:

- индикаторы не строго совпадают с close-only backtest логикой, если та считает только окончательно закрытые свечи.

### 11.3. `TradeBuffer` кодирует нейтральные значения прямо в return values

- пустое окно для `buy_sell_ratio()` даёт `1.0`;
- односторонний поток buy без sell даёт `100.0` (cap, не бесконечность);
- пустое окно для `aggressive_flow()` даёт `0.5`;
- `aggressive_flow()` volume-weighted, а не count-based (Kyle 1985).

Vs предыдущей версии, где `buy_sell_ratio` возвращал `1e6` как sentinel — такое значение могло вызывать blow-up в downstream ML-пайплайне.

### 11.4. Low-level zero-allocation не равен zero-allocation end-to-end

На уровне самого ring buffer решение очень хорошее для latency-sensitive trading system.

Но при анализе production-пути важно не переоценивать это свойство:

- итоговый feature path всё равно делает копии векторов и расчёты под mutex.

### 11.5. У модуля не видно собственных unit-тестов

По текущему поиску по `tests/` прямых тестов на `RingBuffer`, `CandleBuffer` и `TradeBuffer` не видно.

Для production это означает:

- основная верификация идёт косвенно через `FeatureEngine` и интеграционные тесты;
- edge-case контракты buffers живут скорее как «доверяем простоте реализации», чем как формально закреплённая test matrix.

## 12. Краткий итог

Текущий `buffers` — это компактный header-only слой скользящих in-memory окон для свечей и сделок.

Если кратко, модуль делает три вещи:

1. ограничивает историю фиксированным объёмом;
2. сохраняет корректный хронологический порядок при переполнении;
3. адаптирует сырые нормализованные события в форму, удобную для `FeatureEngine`.

На практике это опорный низкоуровневый слой для всего feature-пайплайна:

- `CandleBuffer` кормит теханализ;
- `TradeBuffer` кормит trade-flow микроcтруктуру;
- `RingBuffer` обеспечивает bounded memory и быстрый overwrite.

Модуль несложный по объёму, но критичный по контрактам. Его корректность напрямую влияет на то, какие данные увидят индикаторы, а значит и весь последующий pipeline торгового решения.
